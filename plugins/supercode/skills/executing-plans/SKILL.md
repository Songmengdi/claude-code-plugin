# 执行计划

加载单个计划文件，批判性审查，批量执行任务，通过验收测试。

**开始时声明**: "我正在使用 executing-plans skill 来实施这个计划。"

**核心原则**：带有验收检查点的批量执行。

## 核心原则

1. **批判性审查** — 加载计划后先审查，有问题立即提出
2. **严格遵循** — 按计划步骤执行，使用计划指定的测试方式
3. **批次执行** — 每批次 3 个任务，完成后汇报等待反馈
4. **验收优先** — 通过验收测试后才完成
5. **中文提交** — 使用中文提交信息

## Git 提交策略

- ✅ 验收测试通过后统一提交
- ❌ 不在每个任务完成后频繁提交
- ✅ 提交信息清晰说明实现的功能和通过的测试

---

## 阶段化工作流程

本技能分为 6 个阶段（1-6），每个阶段的具体指令存储在独立文件中。

**当前你只处于一个阶段。禁止猜测其他阶段该做什么。**

| # | 名称 | 详情 |
|---|------|------|
| 1 | 加载和审查计划 | `phases/phase-1.md` |
| 2 | 执行批次 | `phases/phase-2.md` |
| 3 | 汇报 | `phases/phase-3.md` |
| 4 | 继续 | `phases/phase-4.md` |
| 5 | 验收测试和提交 | `phases/phase-5.md` |
| 6 | 完成报告 | `phases/phase-6.md` |

### 执行规则（强制）

1. **每个阶段的第一步必须是 `Read` 对应的 `phases/phase-N.md`**，未读取前禁止执行任何阶段操作
2. 读取后用 `TodoWrite` 创建该阶段任务清单
3. 阶段完成后标记为 completed，**停止并等待用户回应后再进入下一阶段**
4. 禁止跳过阶段，禁止跨阶段执行
5. **严格按计划中的测试方式验证，不要随意测试**
6. **执行测试的顺序**：任务测试 → 验收标准 → 项目测试套件

## 何时停止并寻求帮助

**立即停止执行当：**
- 批次中途遇到阻碍（缺少依赖、测试失败、指令不清楚）
- 计划存在严重缺口导致无法开始
- 你不理解某个指令
- 验收测试反复失败

**要求澄清而不是猜测。**

## 工作流程集成

1. **brainstorming** — 创建 feature 分支和设计文档
2. **writing-plans** — 编写实施计划
3. **executing-plans** — 执行计划（每个阶段调用一次）
4. **code-quality-review** — 代码质量审查（可选）
5. **finishing-a-development-branch** — 合并到基础分支
