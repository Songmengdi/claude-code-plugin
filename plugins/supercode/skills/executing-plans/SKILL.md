# 执行计划（Executing Plans）

## 概述

加载单个计划文件，批判性审查，批量执行任务，通过验收测试后提交代码。

**核心原则：** 带有验收检查点的批量执行。

**开始时声明：** "我正在使用 executing-plans skill 来实施这个计划。"

**Git 提交策略：**
- ✅ 在验收测试通过后统一提交（使用中文提交信息）
- ❌ 不在每个任务完成后频繁提交
- ✅ 提交信息清晰说明实现的功能和通过的测试

## 流程

### 步骤 1：加载和审查计划

**1.1 读取计划文件**

读取计划文件，从 frontmatter 的 `related.design` 读取设计文档（可选）。

**1.2 批判性审查**

- 识别计划中的任何问题或疑虑
- 检查验收标准是否明确、可执行
- **检查每个任务的测试方式是否明确**：
  - 单元测试是否有具体的测试命令？
  - 集成测试是否有具体的测试命令？
  - 用户协助验证是否有详细的验证步骤？
- 如果有疑虑：在开始前与用户提出
- 如果没有疑虑：创建 TodoWrite 并继续

### 步骤 2：执行批次

**默认：前 3 个任务**

对于每个任务：
1. 标记为 in_progress
2. 严格遵循每个步骤（计划包含小步骤）
3. **按计划中提供的测试方式验证**：
   - 如果计划指定了单元测试命令：执行该命令
   - 如果计划指定了集成测试命令：执行该命令
   - 如果计划要求用户协助验证：提示用户进行验证
   - **不要随意测试，必须使用计划中指定的测试方式**
4. 标记为 completed

**重要：不要在每个任务后提交 git**，所有提交将在验收通过后统一进行。

### 步骤 3：汇报

当批次完成时：
- 展示已实现的内容
- **展示按计划中测试方式验证的输出**
- 说："准备接收反馈。"

### 步骤 4：继续

基于反馈：
- 如有需要应用更改
- 执行下一个批次
- 重复直到完成

### 步骤 5：验收测试和提交

所有任务完成后：

**步骤 5.1：运行验收测试**

**首先：执行计划中每个任务指定的测试方式**

检查计划中每个任务的"测试"部分，按顺序执行：
1. **单元测试**：执行计划中指定的单元测试命令
   ```bash
   # 示例：计划中指定的命令
   pnpm test tests/auth/user-auth.test.ts
   ```
   预期结果应与计划中描述的一致

2. **集成测试**：执行计划中指定的集成测试命令
   ```bash
   # 示例：计划中指定的命令
   pnpm test tests/api/auth.integration.test.ts
   ```
   预期结果应与计划中描述的一致

3. **用户协助验证**：提示用户按照计划中的验证步骤进行验证
   ```
   请按照以下步骤验证功能：
   1. 启动服务: `pnpm dev`
   2. 访问: http://localhost:3000
   3. 验证功能是否正常工作
   ```

**然后：执行计划中的整体验收标准**

如果计划头部定义了验收标准，逐一执行：
- ✅ 标记为完成的验收标准
- ❌ 标记为失败的验收标准

**最后：运行项目的测试套件**（可选，根据项目需要）

```bash
npm test / cargo test / pytest / go test ./...
```

**如果任何测试失败：**
- 停止执行
- 报告失败的测试（包括测试输出、错误信息、哪个任务/验收标准失败）
- 等待用户修复指示

**如果所有测试通过：** 继续到步骤 5.2

**步骤 5.2：更新文档状态并提交**

1. 更新 plan 文档的 frontmatter：
   - 将 `status` 从 `plan` 更新为 `executed`

2. 提交所有代码和文档（使用中文提交信息）：

```bash
git add .
git commit -m "feat(<功能名>): 完成功能实现

- 实现了 <具体功能1>
- 实现了 <具体功能2>
- 通过所有验收测试"
```

### 步骤 6：完成报告

```
✅ 计划执行完成！

已实现功能：
- [列出实现的主要功能]

测试执行情况：
- ✅ 任务测试：[X/X] 通过
- ✅ 验收标准：[X/X] 通过
- ✅ 项目测试套件：通过（如执行）

验收结果：
- ✅ 所有测试通过
- ✅ 代码已提交（commit: <hash>）

下一步建议：
- 如果是阶段性计划，执行下一阶段：/supercode:executing-plans <next-phase-plan>
- 或运行 code-quality-review 进行代码质量审查
- 或使用 finishing-a-development-branch 合并到基础分支
```

## 何时停止并寻求帮助

**立即停止执行当：**
- 批次中途遇到阻碍（缺少依赖、测试失败、指令不清楚）
- 计划存在严重缺口导致无法开始
- 你不理解某个指令
- 验收测试反复失败

**要求澄清而不是猜测。**

## 何时重新审视之前的步骤

**返回到审查（步骤 1）当：**
- 合作伙伴根据你的反馈更新了计划
- 基本方法需要重新思考

**不要强行突破阻碍** - 停下来询问。

## 记住

- 批判性审查计划
- 严格遵循计划步骤
- **严格按计划中的测试方式验证，不要随意测试**
- **执行测试的顺序：任务测试 → 验收标准 → 项目测试套件**
- 不要跳过验证
- **不要频繁提交 - 等待验收通过后统一提交**
- **使用中文提交信息**
- 当计划提到时参考 skills
- 批次之间：只需汇报并等待
- 遇到阻碍时停止，不要猜测
- 没有明确的用户同意，永远不要在 main/master 分支上开始实现

## 集成

**必需的工作流程技能：**
- **supercode:writing-plans** - 创建此技能执行的计划
- **supercode:code-quality-review** - 实现完成后的代码质量审查（可选）
- **supercode:finishing-a-development-branch** - 合并到基础分支（可选）

**工作流程：**
1. **brainstorming** - 创建 feature 分支和设计文档
2. **writing-plans** - 编写实施计划（单一或阶段性）
3. **executing-plans** - 执行计划（每个阶段调用一次）
4. **code-quality-review** - 代码质量审查（推荐）
5. **finishing-a-development-branch** - 合并到基础分支（所有阶段完成后）
