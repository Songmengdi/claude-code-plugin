# 阶段 3：并行架构设计

**前提：阶段 2 已完成。**

## 目标

创建详细的实施蓝图。

## 执行方式

### 并行启动 2 个 code-architect 子代理

使用 `Agent` 工具并行调用，提供不同的设计视角：

```yaml
subagent_type: "supercode:code-architect"
description: "设计实施方案"
prompt: |
  设计功能实施方案：

  **参考文档**：
  - 设计文档：docs/plans/YYYY-MM-DD-<feature-name>-design.md

  **任务信息**：
  - 规模：[小型/中型/大型]
  - 视角：[干净实现 | 务实平衡]

  **代码上下文**：
  - 需修改的文件：[阶段2探索得到]
  - 现有代码结构：[阶段2分析结果]

  输出详细的实施蓝图，包含：
  1. 阶段划分（如需）
  2. 每个任务的文件清单和修改内容
  3. 每个阶段的验收标准和测试策略
```

## 子代理输出要求

### 阶段划分（中/大型设计）

- 每个阶段有明确的功能边界
- 每个阶段是闭环的、可独立验证的
- 阶段间依赖清晰

### 任务分解

- 每个任务是一个完整的、不破坏代码的最小变更
- 明确要创建/修改的文件和位置
- 说明实现的核心思路

### 验收标准（关键）

每个阶段/任务必须包含**可衡量的验收标准**：

| 阶段类型 | 测试策略 | 示例 |
|---------|---------|------|
| 纯逻辑实现 | 单元测试 | 函数输入输出验证 |
| 对接外部服务 | 集成测试 | API 调用验证 |
| UI/交互功能 | 用户协助验证 | 手动操作步骤 |


**验收标准必须**：
- 具体、可执行
- 有明确的测试命令或验证步骤
- 有清晰的预期结果
