# 阶段 3：任务分解

**前提：阶段 2 已完成。**

## 目标

基于设计文档创建详细的任务分解。

**输入**：已批准的设计文档（包含架构决策）
**输出**：任务清单、阶段划分（如需）、验收标准

## 执行方式

### 1. 评估任务规模

根据设计文档判断：
- **小型**：≤5 个核心任务 → 单一计划
- **中型**：6-10 个核心任务 → 2-3 个阶段
- **大型**：>10 个核心任务 → 多个独立阶段

### 2. 创建任务分解

**任务设计原则**：
- 每个任务是一个完整的、不破坏代码的最小变更（5-15 分钟）
- 每个任务明确：文件路径、修改类型、修改位置、实现要点

**任务必须包含参考信息**（重要）：
- **参考文档**：链接到设计文档、API 文档等
- **参考代码**：链接到类似的实现或相关代码片段
- **类型定义**：直接给出类型定义或明确指出查找位置
- **防止 AI 猜测**：所有关键信息都有明确来源

**参考链接格式**：
```markdown
**参考文档：**
- 设计文档：[docs/plans/...-design.md](docs/plans/...-design.md)
- API 文档：[path/to/api.md](path/to/api.md)

**参考代码：**
- 类似实现：[src/path/file.ts:行号范围](src/path/file.ts#L行号)
- 接口定义：[src/types/file.ts:行号范围](src/types/file.ts#L行号)

**类型定义：**
- 方式1：直接给出（新类型）
- 方式2：参考现有（链接到定义位置）
```

**阶段划分原则**（中/大型设计）：
- 每个阶段有明确的功能边界
- 每个阶段是闭环的、可独立验证的
- 阶段间依赖清晰

### 3. 定义验收标准

每个阶段/任务必须包含**可衡量的验收标准**：

| 阶段类型 | 测试策略 | 示例 |
|---------|---------|------|
| 纯逻辑实现 | 单元测试 | 函数输入输出验证 |
| 对接外部服务 | 集成测试 | API 调用验证 |
| UI/交互功能 | 用户协助验证 | 手动操作步骤 |

**验收标准必须**：
- 具体、可执行
- 有明确的测试命令或验证步骤
- 有清晰的预期结果
