# 阶段 3：文档结构设计

前提：阶段 2 已完成，系统模型和内容清单已确认。

## 目标

设计以**可执行场景**为主干的文档导航体系。

---

## 设计原则

**AI-First 原则**：AI 最好理解的是"生产可用的最小 demo"，而非概念解释。

- 文档从"我想做 X"的场景出发
- 每个文档围绕一个可执行闭环组织
- 概念解释是示例的附属，不是主体

---

## 3.1 文档拆分策略

**按场景/任务拆分，而非按模块/概念拆分**

| 拆分维度 | 错误（说明书思维） | 正确（场景思维） |
|----------|-------------------|------------------|
| 入口文档 | "核心概念" | "5 分钟完成 X" |
| API 文档 | "API 参考" | "如何调用 Y 接口" |
| 错误文档 | "错误码表" | "遇到 Z 错误怎么办" |

### 拆分算法

```
1. 从阶段 2 的"可文档化内容清单"中提取用户任务
2. 每个任务 = 一个文档
3. 任务粒度：
   - 一个任务 = 一个可执行闭环
   - 用户看完能直接动手
4. 按任务依赖排序：前置任务 → 主任务 → 进阶任务
```

### 示例

```
❌ 按模块拆分（说明书）：
- concepts/core.md — 核心概念
- concepts/engine.md — 引擎概念
- api/create.md — create API
- api/run.md — run API
- errors.md — 错误码表

✓ 按场景拆分（AI-First）：
- quickstart.md — 5 分钟完成一个完整调用
- create-instance.md — 如何创建实例（含参数选择、默认行为）
- run-task.md — 如何执行任务（含状态跟踪、超时处理）
- handle-errors.md — 遇到错误怎么办（按错误场景组织）
- advanced/custom-config.md — 需要定制化时
```

## 3.2 单文档结构

**结构：场景 → 示例 → 参数 → 失败路径**

```markdown
# [场景名称]

## 我要做什么
[一句话描述用户想完成的任务]

## 最小示例
[可直接复制执行的完整代码]

初始状态：[调用前需要什么]
调用代码：[完整代码]
返回结果：[实际输出]
状态变化：[调用后系统变成什么]

## 参数说明
[示例中涉及的参数，每项附 source]

## 可能的失败
[示例执行过程中可能遇到的错误，按场景组织]

触发条件：什么输入/状态会触发
后果：抛什么错/进入什么状态
如何恢复：[可执行的恢复步骤]

## 下一步
[相关场景链接]
```

## 3.3 index.md 设计

index.md 是**任务地图**，不是概念目录。

```markdown
# [项目名] 任务地图

## 我想...

- [5 分钟快速上手](quickstart.md) — 完成第一个调用
- [创建一个实例](create-instance.md) — 配置参数、理解默认行为
- [执行任务并跟踪状态](run-task.md) — 超时处理、状态轮询
- [处理错误](handle-errors.md) — 遇到 X/Y/Z 错误怎么办

## 深入主题

- [自定义配置](advanced/custom-config.md) — 默认配置不满足时
- [状态机详解](advanced/state-machine.md) — 需要理解内部状态流转

## 索引

- [参数完整参考](reference/params.md)
- [错误完整参考](reference/errors.md)
```

**规则**：
- 入口必须是"我想..."的任务列表
- 每个链接说明"什么时候需要看"
- 概念/参考文档放在"索引"区域，不是主体

---

## 3.4 规模约束

- 单文档硬性上限：1000 行
- 单文档预警阈值：800 行
- 超过阈值时：按场景进一步拆分（不是按长度切分）
