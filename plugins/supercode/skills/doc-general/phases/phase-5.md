# 阶段 5：并行验证

前提：阶段 4 已完成，所有文档已生成。

## 目标

分组并行验证文档的正确性和结构质量。

---

## Task 分组策略

**核心原则**：按文档数量均分，每个 Task 验证一组文档。

### 分组算法

```
1. 统计待验证文档总数 M
2. 设每组文档数上限 G = ceil(M / 3)（保证最多 3 个 Task）
3. 按顺序切分：[1..G], [G+1..2G], [2G+1..M]
```

---

## 并行控制

- **最多 3 个 Task 并行**

---

## 单个 Task 的 prompt 结构

### 1. 任务定义

```
你的任务：验证以下文档的正确性和结构质量

| 文档路径 |
|----------|
| concepts/core.md |
| guides/quickstart.md |
| api/main-api.md |
```

### 2. 验证规则

```
【source 标记验证】
1. 提取文档中所有 `source: source/{类型}/{路径}:行号` 标记
2. 逐一验证：
   - 文件存在
   - 行号在文件范围内
   - 源码语义与文档描述一致

【链接验证】
1. 提取文档中所有相对路径链接 `[文字](path/to/file.md)`
2. 逐一验证：目标文件存在

【示例完整性】
- 每个接口/功能是否有可执行示例？
- 示例是否是闭环（初始状态 → 完整调用 → 返回结果 → 状态变化）？
- 是否存在伪代码、不完整 JSON、未定义占位符、省略号？

【解释支撑】
- 是否存在无示例支撑的参数/行为描述？
- 解释是否被示例证明？

【状态机完整性】（仅对有状态模块）
- 是否描述了可能的状态？
- 是否描述了状态转换条件？
- 是否描述了非法状态/转换的后果？

【失败地图完整性】
- 错误/约束是否与接口同级出现？
- 每个错误是否有：触发条件、后果、是否可恢复？

【自洽性】
- 是否假设了外部上下文？
- 是否引用了未定义的概念？

【用词规范】
- 是否包含模糊词："通常""一般""建议""可能"？
```

### 3. 输出格式

```
如果全部通过，输出：✓ 全部验证通过

如果有问题，按以下格式输出：

## source 不一致

### concepts/core.md
- [文档内容] `source: source/github/core/engine.ts:45`
  - 源码实际：[真实行为]

## 示例缺失

### api/main-api.md
- 接口 `createEngine` 无可执行示例
- 参数 `timeout` 的默认值描述无示例证明

## 示例不完整

### guides/quickstart.md
- 示例缺少"初始状态"描述
- 示例使用省略号：`{ ... }`

## 状态机缺失

### concepts/core.md
- 模块有状态但未描述状态转换条件
- 未描述非法状态的后果

## 失败地图缺失

### api/main-api.md
- 错误 `TimeoutError` 未说明可恢复性
- 错误未与接口同级出现

## 自洽性问题

### concepts/core.md
- 引用了未定义的概念 "ExecutionContext"
- 假设读者了解 "Promise" 的行为

## 模糊用词

### guides/quickstart.md
- "通常建议设置 timeout"
- "一般返回 JSON 格式"

## 失效链接

### api/main-api.md
- `[配置说明](guides/config.md)` — 目标文件不存在
```

**约束**：只输出有问题的项，无问题的文档不出现在报告中。

---

## 执行流程

1. **分组**：按数量均分，最多 3 组
2. **并行验证**：启动 Task，每组一个
3. **汇总问题**：收集所有 Task 输出
4. **修正文档**：根据问题列表修正
5. **回归验证**：修正部分重新验证（可迭代）
6. **完成**：全部通过后结束

---

## 本阶段输出

- 问题列表（按类型分组）
- 修正后的文档（如有问题）

## 完成条件

所有验证项通过，用户确认后任务结束。
